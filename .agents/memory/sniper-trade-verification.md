## The rule

Every `SniperPosition` has two separate tx signature fields:
- `entrySig` — the buy tx, set at `enterPosition()`, **never overwritten**
- `exitSig` — the sell tx, set only after a **confirmed** on-chain sell (in `closePosition` after the delete guard fires, and in `partialClose` when `exitTxSig` is non-empty)

A closed position where `exitSig` is `undefined`/`null` is **unverified** — its P&L may be fake (generated by the old buggy code that recorded closes before the sell actually happened).

**Why:** The old code pre-deleted positions from `openPositions` and sent Telegram before the Jupiter swap confirmed. If the swap failed silently, the position disappeared from the UI and a closed record was written with fabricated P&L. The `exitSig` field makes this auditable.

**How to apply:**
- `txSignature` still stores the latest tx (buy on entry, overwritten with sell on close) for backward compat — don't rely on it to identify entry vs exit.
- `entrySig` is always the buy tx; `exitSig` is only the sell tx.
- The UI shows ⚠️ Unverified badge on any history entry where `!pos.exitSig`.
- `purgeUnverifiedHistory()` (service) + `POST /api/sniper/history/purge-unverified` (route) + `usePurgeUnverifiedHistory()` (React hook) let the user bulk-delete all unverified history.
- DB columns: `entry_sig TEXT DEFAULT ''` and `exit_sig TEXT` (nullable). Historical rows loaded before this migration have `exit_sig = NULL` and correctly show as unverified.
